Schedule Counter — Setup Guide

From zero to a working, self-updating labor tool. No coding experience needed — you copy files, set one setting, run two functions, deploy, and connect your sales data. Follow the steps in order and you'll be done in about 30 minutes.

⏱ About 30 minutes

Schedule Counter takes your published schedule and lays it against your sales, hour by hour, so you can see exactly where you're heavy, thin, or on target. It lives inside one Google Sheet — there are no IDs, URLs, or passwords to paste anywhere.

How the whole thing fits together (worth understanding before you start):

  • The Sheet is the database. The tool's tabs live here.
  • Sheet1 is the landing zone where your CFA sales data shows up (you'll connect that in Step 9).
  • The web app is the screen you and your team actually use.
  • Three weekly jobs run on their own — pulling in sales, updating the forecast, and reminding you to post the schedule.
⚠️ The one rule that keeps it working: never create or rename a tab by hand. The setup function builds every tab with the exact name and headers the tool expects. Make a tab called "Config" instead of config, or rename one later, and the tool quietly stops working with no obvious error.

Before You Start

  • A Google account that can create Google Sheets — your CFA Workspace account is ideal.
  • The project files from this folder (all the .gs and .html files).
  • Access to your store's CFA Home → Analytics Hub (ThoughtSpot) sales data — you'll wire that up in Step 9.
  • About 30 minutes, on a computer (not a phone).
1

Create the Google Sheet

  1. Go to sheets.google.com and click Blank spreadsheet.
  2. Rename it something clear, like "Schedule Counter — [Your Store]" (click the title, top-left).
  3. Leave the default Sheet1 tab exactly as it is — don't rename or delete it. It becomes the landing zone for your sales data later.

This single spreadsheet is both your database and your sales-sync target. Everything the tool does happens inside it.

2

Open the Apps Script Editor

  1. In your new Sheet, click Extensions → Apps Script. A code editor opens in a new browser tab.
  2. You'll see a starter file called Code.gs with an empty myFunction.

Because you opened the editor from inside the Sheet, the script is bound to that Sheet. That binding is why you never paste a Sheet ID — the code already knows which spreadsheet it belongs to.

3

Paste the Files

For each file in this project, create a matching file in the editor and paste its contents. The app loads files by name, so names must match exactly, capitalization included.

  • Script files (e.g. Code.gs): click + → Script, name it to match, paste, save.
  • HTML files (Index, JavaScript, Stylesheet, Onboarding, TrackerEmbed, ProductivityTracker): click + → HTML, name it to match (no .html in the name), paste, save.

Replace the contents of the starter Code.gs with the project's Code.gs (don't make a second one). Delete the empty myFunction if it's in your way. Press Ctrl/Cmd+S to save everything.

Why exact names matter: the main page is loaded as Index and pulls in JavaScript, Stylesheet, and the others by name. A lowercase index or a typo'd Stylesheet shows up later as a blank page or missing styling, not an error message.
4

Set the One Required Setting

The tool emails you if one of its weekly background jobs ever fails — that's how you find out the sales sync didn't run before it throws off your numbers. Tell it where to send that alert:

  1. In the Apps Script editor, click ⚙️ Project Settings (the gear in the left sidebar).
  2. Scroll down to Script Properties and click Add script property.
  3. Property: ALERT_EMAIL  ·  Value: the email address that should get failure alerts.
  4. Click Save script properties.
Skipping this won't stop the app from working, but you'll be flying blind if a weekly job fails silently. Take the 30 seconds and set it.
5

Build the Tabs

  1. At the top of the editor, open the function dropdown (it may say doGet) and choose initSheets.
  2. Click ▶ Run.
  3. The first run triggers Google's permission screen: Review permissions → pick your account → Advanced → Go to (project name) → Allow. This is normal for your own scripts.
  4. Open the Execution log at the bottom. When it finishes you'll see: 
    initSheets complete. All tabs verified.

This one function builds every tab the tool needs and seeds your default settings and sales curves. You'll see these tabs appear in the Sheet:

TabWhat it holds
configAll your settings — SPLH goals, labor shares, email, location. The one tab you'll edit by hand.
sales_curvesThe hourly weight distribution per day used to spread sales across the day.
sales_historyHourly sales per day per week, kept for forecasting.
schedule_historyHourly staffing (FOH/BOH) per day per week.
weekly_summaryOne row per week — SPLH, peak hour, goal %.
productivity_trackerLong-term monthly productivity scores.
app_cachePre-computed data for fast app loads. Never edit by hand.
Sheet1The landing zone your CFA sales data drops into (Step 9).
Safe to re-run. initSheets skips any tab that already exists and never deletes data. If you ever think a tab got messed up, re-running this is the safe first move.
6

Install the Background Jobs

This is a separate step from Step 5 — building the tabs does not install the automation. You must run this too, or the weekly numbers will never update on their own.
  1. In the function dropdown, choose installTriggers.
  2. Click ▶ Run.
  3. Confirm it worked: click the clock icon (Triggers) in the left sidebar. You should see three time-driven triggers listed.

Those three jobs are the heart of the automation:

WhenJobWhat it does
Sunday 10 PMSync checkConfirms your CFA sales data actually landed in Sheet1. Emails you if the sync failed, so you can fix it before Monday.
Monday 6 AMWeekly pipelineArchives the week's sales, updates the curves with smoothing, runs bias detection, cleans old data, and rebuilds the cache.
Thursday 7 PMSchedule reminderEmails you a reminder (with last week's performance) to post the new schedule.

To reinstall these later, run installTriggers again. To remove them all, run deleteAllTriggers.

7

Deploy as a Web App

This turns the script into a real web page with a link your team can open.

  1. Click Deploy → New deployment (top-right).
  2. Click the gear ⚙ next to "Select type" and choose Web app.
  3. Fill in:
    • Description: Schedule Counter (anything works)
    • Execute as: Me
    • Who has access: Anyone within [your domain] — keeps it inside your CFA Workspace
  4. Click Deploy, authorize if asked, and copy the Web app URL. That link is the tool — bookmark it.
Updating later: any time you change the code, you must redeploy or nothing changes for your users. Go to Deploy → Manage deployments → ✎ Edit → Version: New version → Deploy. The URL stays the same every time.
8

First-Run Onboarding

Open your Web app URL. The first time it loads, a built-in onboarding wizard walks you through the settings that make the numbers yours:

  • Location name and which dayparts you run.
  • SPLH goals — overall, FOH, and BOH targets.
  • Labor shares — how you split staffing between front and back of house.

Everything the wizard sets is written to the config tab, so you can come back and adjust it any time in the Sheet — you never touch code to change a setting.

First load is slow (10–20 seconds) while Apps Script wakes up. That's normal. If you ever get a blank white page instead, see Troubleshooting.
9

Connect Automatic Sales Data (ThoughtSpot / Analytics Hub)

This is the piece that makes the tool run itself. You're going to set up a weekly sync that pulls your store's hourly sales out of CFA's analytics and drops them into the Sheet1 tab every Sunday night — right before the tool's own Sunday 10 PM check looks for them. It's the most involved step, but you only do it once.

Part A — Build the search

  1. In CFA Home, go to Analytics Hub → Custom Reports → ThoughtSpot Search and open the Sales Analytics search.
  2. Build the search so it includes these, as columns and filters (your tokens across the top should read like this):
    • Sales (the sales measure)
    • Hour (24 hr)
    • Business Date
    • time filter: this week
    • sort by Business Date, and sort by Hour (24 hr)
    • store filter: your restaurant by its number — all sales channels (don't restrict it to drive-thru only)

Part B — Add the "Today date" formula

  1. In the left column panel, click + Add → Formula.
  2. Name it Today date, type today() into the formula editor (it'll say "Formula is ready to save"), and click Save.
  3. Make sure its checkbox is ticked so it's included, along with Hour (24 hr), Business Date, and the Sales measure.
The today() formula stamps every sync with the date it ran — a simple way to confirm at a glance that fresh data came through.

Part C — Set up the weekly sync to your Sheet

  1. Up top, click the … (more) menu → Sync → Sync to Sheets. The first time, you'll be asked to link your Google account — use the same account that owns your scheduling Sheet.
  2. In the pipeline editor, fill in:
    • Pipeline name: something like PL-Last week sales by day and hour
    • File name: your scheduling Google Sheet (the one from Step 1)
    • Sheet name: Sheet1
    • Cell: A1
  3. Check Sync schedule and set Update occurs every Week, at 21:00 (9 PM), your timezone (America/Chicago for Cockrell Hill), on Sunday (SU).
  4. Click Save.
The 9 PM Sunday time matters. The sync must land before the tool's Sunday 10 PM check and Monday 6 AM pipeline. 9 PM Sunday gives it an hour of cushion. If you move it, keep it Sunday evening, before 10 PM.

What you've built: every Sunday at 9 PM, this week's sales by day and hour overwrite Sheet1 starting at A1. The tool reads the Hour, Business Date, and sales columns by name (any order), then clears Sheet1 after the Monday run so it's fresh for the next sync.

What your Sheet1 row 1 should look like after the first sync: 
A: Hour (24 hr)   B: Business Date   C: Today date   D: Total Sales
The tool reads the Hour, Business Date, and Total Sales columns by name, so the order doesn't matter and the extra Today date column is fine. If you ever get a "Sheet1 missing required columns" error, compare your headers against this.
10

Backfill & Verify

Once sales data is landing in Sheet1, prime the history so the forecast has something to learn from, then confirm everything's talking:

  1. In the app, go to Settings → Automation and click Backfill History (this reads what's in Sheet1 into your history), then Rebuild Curves.
  2. Open the Schedule Analysis tab in the app and upload a schedule export — you should get a color-coded staffing grid with the sales overlay.
  3. Check the config tab in the Sheet shows your store name and goals from onboarding.
✅ A populated staffing grid, history in your sales_history tab, and three triggers under the clock icon means you're fully live.

How the Weekly Automation Works

Once it's set up, you don't touch it — but here's the cycle so you know what's happening behind the scenes and what each alert means:

  • Through the week — your sales data flows into Sheet1 automatically from ThoughtSpot (Step 9).
  • Sunday 10 PM — the sync check makes sure that data actually arrived. If Sheet1 is empty or stale, you get an email so you can fix it before the pipeline runs.
  • Monday 6 AM — the pipeline archives the week's numbers, re-smooths your sales curves, checks the forecast for bias, cleans up old rows, clears Sheet1, and rebuilds the cache so the app loads fast.
  • Thursday 7 PM — you get a reminder (with last week's performance) to post the new schedule.

Where Your Team Opens It

It's a web app, so it's just a link — no install. Open the Web app URL from Step 7 and:

  • Bookmark it on the office computer.
  • Add it to a tablet's home screen if you want it one tap away on the floor.
  • Share the link with whoever builds the schedule — access is already limited to your domain from the deploy settings.

Troubleshooting

Blank white screen when you open the web app — almost always the browser libraries. They're version-pinned in this copy; if you ever pasted an older Index, confirm the React and Babel <script> tags carry specific versions (react@18.3.1, @babel/standalone@7.26.4). An unpinned Babel floats to v8 and white-screens the page. After fixing, redeploy a new version and hard-refresh (Ctrl/Cmd+Shift+R).

"Tab "X" not found. Run initSheets() first." — you used or deployed the app before running initSheets (Step 5), or a tab got renamed/deleted. Run initSheets from the editor and try again.

"app_cache tab not found" or "sales_curves is empty" — same fix: run initSheets (Step 5). It creates and seeds both.

You got a Sunday "sync may have failed" email — the sales data didn't reach Sheet1. Check your ThoughtSpot export (Step 9) actually ran and that Sheet1 has recent rows with a current Business Date.

"Sheet1 missing required columns" — the data that landed doesn't have headers containing Hour, Business Date, and Total Sales. Fix the export so those columns come through with those names.

The weekly numbers never update on their own — the triggers aren't installed. Run installTriggers (Step 6) and confirm three appear under the clock icon.

"You do not have permission to call …" — re-authorize: run initSheets once from the editor and click Allow.
